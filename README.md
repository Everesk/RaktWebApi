## Описание проекта

Простое ASP.NET Core Web API приложение для управления событиями.

Проект демонстрирует:
- базовую архитектуру Web API
- разделение слоёв
- валидацию данных
- использование Dependency Injection
- работу со Swagger
- глобальную обработку ошибок
- логирование через Serilog

Приложение позволяет:
- получать список всех событий
- получать событие по идентификатору
- создавать новое событие
- обновлять существующее событие
- удалять событие

Данные хранятся в памяти приложения (без базы данных).

## Стек

- .NET 9
- ASP.NET Core Web API
- Swagger
- Serilog

## Структура проекта

- Controllers — HTTP-эндпоинты
- Models — сущности и DTO
- Services — бизнес-логика
- Mappers — преобразование DTO и сущностей
- Extensions — конфигурация приложения
- Common
  - Exceptions — пользовательские исключения
  - Middleware — глобальная обработка ошибок
  - Helpers — формирование ProblemDetails

## Запуск проекта

Клонировать репозиторий:

git clone <ссылка-на-репозиторий>

Перейти в папку проекта:

cd RaktWebApi/RaktWebApi

Запустить приложение:

Только http:
dotnet run

http и https:
dotnet run --launch-profile https

После запуска в консоли будут выведены адреса:

http://localhost:5007  
https://localhost:7130  

## Swagger

Swagger доступен только в режиме разработки по адресу:

/swagger

## API

GET /events  
Получить список всех событий  
Ответ: 200 OK

---

GET /events/{id}  
Получить событие по идентификатору  
Ответы:
- 200 OK — если найдено
- 404 Not Found — если не найдено

---

POST /events  
Создать новое событие  
Content-Type: application/json

Пример тела:

{
  "title": "Встреча команды",
  "description": "Обсуждение задач",
  "startAt": "2026-03-25T10:00:00",
  "endAt": "2026-03-25T11:00:00"
}

Ответ: 201 Created

---

PUT /events/{id}  
Обновить событие  

Ответы:
- 204 No Content — успешно
- 404 Not Found — если не найдено

---

DELETE /events/{id}  
Удалить событие  

Ответы:
- 204 No Content — успешно
- 404 Not Found — если не найдено

## Валидация

Реализованы следующие проверки:

- Title обязателен
- StartAt обязателен
- EndAt обязателен
- EndAt должен быть позже StartAt

При ошибке возвращается 400 Bad Request.

## Глобальная обработка ошибок

В приложении реализована глобальная обработка исключений через middleware.

Все необработанные исключения:
- перехватываются на уровне pipeline
- логируются с использованием Serilog
- возвращаются клиенту в формате ProblemDetails (RFC 7807)

Формат ответа содержит:
- HTTP-статус
- краткое описание ошибки (title)
- подробности (detail)
- путь запроса (instance)
- идентификатор запроса (traceId)

Пример ответа:

{
  "title": "Ресурс не найден",
  "status": 404,
  "detail": "Событие с идентификатором '...' не найдено.",
  "instance": "/events/...",
  "traceId": "..."
}

Обработка ошибок:

- ValidationException → 400 Bad Request
- NotFoundException → 404 Not Found
- прочие исключения → 500 Internal Server Error

## Обработка статус-кодов

Для ситуаций без исключений (например, неверный маршрут или HTTP-метод) используется middleware обработки статус-кодов.

Покрываются:
- 404 Not Found
- 405 Method Not Allowed

## Логирование

В приложении используется Serilog.

Логи записываются:
- в консоль
- в файл logs/log-*.txt (с ежедневной ротацией)

Особенности:
- ошибки уровня 500 логируются с полным stack trace
- ошибки уровня 400/404 логируются без stack trace
- каждый ответ содержит traceId для сопоставления с логами

## Отладка (Visual Studio)

Visual Studio может останавливать выполнение на строке throw, даже если исключение обрабатывается middleware.

Чтобы отключить это для ожидаемых ошибок:

Отладка → Окна → Параметры исключений (Ctrl + Alt + E)

Добавить исключение:

RaktWebApi.Common.Exceptions.NotFoundException

и снять флаг "Выбрасываемые".
